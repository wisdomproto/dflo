# 임상 지식 브레인 (Clinical RAG Knowledge Brain) — 설계 스펙

> 상태: 설계(브레인스토밍 결과). 구현 계획(plan)은 별도. 2026-06-06.

## 1. 비전 / 목표

연세새봄의원 187 성장클리닉의 **실제 진료 데이터 + 국제 논문**을 융합한 "진짜 의사처럼 똑똑한" RAG 지식 브레인을 만든다. 하나의 지식 브레인을 만들고 **세 개의 출력 표면(입)** 이 그 뇌를 빌려 쓴다.

**핵심 원칙: 파인튜닝(학습)이 아니라 RAG 참조(retrieval-grounding).**
- 진료 데이터를 모델 가중치에 굽지 않는다. 임베딩해 DB에 두고 질의 시 관련 항목만 꺼내 프롬프트에 참조시킨다.
- 이유: (1) 개인정보 — PHI가 가중치에 안 박혀 환자 삭제요청 시 DB에서 제거하면 끝(PIPA 삭제권), (2) 항상 최신 — 새 진료 임베딩 추가 = 즉시 반영(재학습 불필요, "진료할수록 똑똑해지는 루프"), (3) 추적 가능 — 답변의 근거(케이스·논문) 인용 표시 가능.

## 2. 우선순위 (출력 표면 = "입")

| 순위 | 표면 | 데이터 접근 | 규제 강도 |
|---|---|---|---|
| 1 | **마케팅 콘텐츠 생성** | 비식별·집계 | 높음 (의료광고법) |
| 2 | **환자 상담 챗봇** | 비식별·교육 | 높음 (무면허 의료행위/의료기기) |
| 3 | **원장 진료 의사결정 보조** | 원본 PHI | 중간 (의사 루프 안) |

> 주의: 1·2순위가 공개·환자 대상이라 규제가 빡센 쪽. 따라서 **가드레일 + 비식별 레이어를 "먼저" 만든다.** 우선순위와 안전설계 순서가 일치.

## 3. 지식 소스 (공유 브레인의 재료)

### 3-1. 논문 라이브러리
- **국제 논문(PubMed) 메인.** NCBI E-utilities(esearch→efetch)로 **초록(abstract)만** 수집 — 그게 사실상 전부.
- 영어 초록은 언어 중립 → 같은 라이브러리로 ko/th/vi/en 콘텐츠 전부 grounding. (한국 논문은 ko 전용 소수 보강)
- **큐레이션 미리 적재(pre-load).** 주제축이 안정적: 성장호르몬·뼈나이·성조숙증·비만·알러지/만성염증·수면·영양·갑상선 등. 한 번 모아 임베딩, 가끔 갱신.
- **인종(population) 태그 — 신뢰도 등급제:**
  - 명시(explicit): 초록에 "Korean children" 등 직접 언급 → 세분 태그(Korean/Thai/...)
  - 추정(inferred): 저자 소속국·저널·MeSH("Asian People", 국가 MeSH)로 유추
  - 불명(unknown): 단서 없음 → 정직하게 Unknown
  - 전략: **자동은 거칠게(Asian/SEA/Caucasian/Unknown), 손질은 수치 민감 핵심 논문만** 큐레이션 단계에서 수동 정확화.

### 3-2. 비식별 임상 인사이트 레이어 (1·2순위 공통분모)
실제 진료 데이터에서 **비식별·집계** 형태로 파생. 공개 표면(마케팅·챗봇)은 원본 환자 RAG가 아니라 이 레이어를 쓴다.
- (a) **집계 통계** — "이 프로필 N명, 평균 관리기간·전형적 검사 흐름·반응 분포" (숫자, 안전, 신뢰의 뼈대)
- (b) **합성(composite) 케이스** — 집계 패턴에서 만든 *전형적 사례* ("8세 여아, 뼈나이 1년 지연, 보통 이런 흐름"). **실존 개인 익명화가 아니라 합성** → 경험담도 아니고 재식별 0. (실제 개별 케이스 디테일은 #3 원장 모드에서만)
- 파생 출처: 기존 `patientCategories`(8분류) + `patient_analyses`(response_level·treatment_phase·growth_metrics·sub_categories·risk_flags).
- **안전장치:**
  - **k-익명성 임계** — 유사 환자 N명 미만 희귀 패턴은 통계·케이스로 내보내지 않음(재식별·과장 방지).
  - **원장 컨펌** — 합성 케이스/통계도 공개 전 검수(기존 컨펌 워크플로우 재사용).

### 3-3. 원본 임상 임베딩 (3순위 전용, PHI)
- 기존 Phase 21 `patient_embeddings`(pgvector) + `match_patient_embeddings` RPC = 이미 retrieval 방식. 출발선이 RAG.
- 원장 모드에서만 원본 PHI 참조. 공개 표면 차단.

## 4. 주장(claim) 분류 — 인종 처리의 핵심
- **메커니즘·생물학** (성장호르몬 작용, 수면-GH, 영양결핍→성장판): **인종 무관** → 양질 논문 아무거나 OK.
- **수치·표준·결과** (키 백분위, 뼈나이 표준, 사춘기 평균연령, 치료 반응률): **인종 민감** → **우리 임상 데이터(=population-matched)가 주도**, 논문은 보조.
- 효과: 논문 인종 태그가 불완전해도 치명적이지 않음(수치는 임상이 담당). 이미 계산기에서 한국 LMS / 태국 TSPE 2022로 분리해둔 철학과 동일.

## 5. 출력 어댑터 (입 3개) + 컴플라이언스

### 5-1. 마케팅 (1순위)
- 비식별 인사이트 + 논문 retrieval → 프롬프트 grounding → 콘텐츠 생성. 기존 `marketing_config`(브랜드 보이스 + bannedKeywords) 결합.
- **의료광고법 가드레일**: 효과 단정·치료경험담·과장 차단. 논문은 "교육 콘텐츠"를 강화하지 "우리 치료가 N cm 키운다" 효과광고를 만들지 않음.

### 5-2. 환자 챗봇 (2순위)
- 교육·일반정보 + **상담 예약 유도**만. **개별 진단·처방 절대 금지**(규제선). "정보 제공/상담 도우미"이지 "의료 AI"가 아님.

### 5-3. 원장 진료 보조 (3순위)
- 시나리오: 새 환자 피검사·알러지 검사 결과 입력 → 유사 검사 프로필 과거 환자 검색 → 그들의 처방 + 결과(반응도·PAH 변화) + 관련 논문 → **근거 달린 추천안** 제시.
- **반드시: 추천 ≠ 자동 처방. 원장이 루프 안에서 최종 결정.** "이 약 N mg 처방하세요"(단정형, SaMD 분류 위험)가 아니라 **"유사 케이스 N명은 이렇게 했고 결과는 이랬다 + 논문 근거 + 신뢰도"** 제시형.
- **신뢰도 표시 필수** — 희귀 프로필은 유사 케이스 적어 신뢰 낮음 → 확신도·근거 함께 노출. 블랙박스 금지.

## 6. 공통 가드레일 게이트
- 의료광고법(효과 단정·경험담·과장 금지, `bannedKeywords`)
- **인구 불명/비아시아 근거 플래그** — 수치 주장이 비아시아·인구불명 논문에만 기대면 경고.
- **개별 의료행위 금지**(챗봇) / **의사 최종결정**(원장 모드).
- **인용 + 신뢰도 표시** — 근거(케이스/논문)와 확신도 항상 노출.
- **k-익명성** — 희귀 패턴 비공개.

## 7. 재사용 가능한 기존 자산
- `patient_embeddings`(pgvector) + `match_patient_embeddings` RPC (Phase 21) — RAG 인프라.
- `embedder.ts`(환자 정규화 텍스트→임베딩), `coachingGenerator.ts` (ai-server).
- `patient_analyses`: response_level / treatment_phase / growth_metrics / sub_categories / risk_flags (Phase 17).
- `patientCategories.ts` 8분류 (Pure 함수).
- `lab_tests`(IgG4·MAST·유기산 등 panel_type), 알러지 데이터, `medications`(MED/INJ/PRO) + `prescriptions`.
- `marketing_config`(브랜드 보이스 + bannedKeywords), 원장 컨펌 워크플로우.

## 8. 단계별 빌드 순서 (제안)
- **A. 기반** — 논문 라이브러리(수집·인종태그·임베딩) + 비식별 임상 인사이트 레이어(집계통계 + 합성케이스 + k-익명성). → 1·2순위 동시 커버.
- **B. 마케팅 RAG 생성** — 콘텐츠 생성에 retrieval grounding + 의료광고법 가드레일 결합 (기존 콘텐츠 스튜디오에 연결).
- **C. 환자 챗봇** — 교육 + 예약 유도, 개별진단 차단.
- **D. 원장 진료 보조** — 원본 PHI 모드: 검사결과 → 유사케이스+처방+결과+논문 → 추천안(의사 루프).

## 8-1. 필수 보조 자산 — 약물 코드 사전 (legend)
`medications` 테이블 이름은 **병원 내부 코드명**이라 약의 정체가 문자열만으론 안 드러난다. RAG가 코드명만 보면 오판한다(프로토타입에서 **에이큐_G = 성장호르몬**을 "성장 보조제"로 오분류).
→ **코드 → 실제 약물명/계열/적응증 사전**을 만들어 retrieval·프롬프트에 같이 주입해야 한다. 성장치료 3대 축 예시: 성장호르몬(에이큐_G 등) / GnRH agonist(루프린) / aromatase inhibitor(아리미덱스). 원장 큐레이션으로 자주 쓰는 약부터 매핑.

## 8-2. 프로토타입 검증 결과 (2026-06-06)
실제 환자(홍준서, 10.5세 남, 49회 진료)로 #3 원장 처방보조를 검증. 초진 혈액검사(Estradiol·LH·FSH·IGF-1·VitD 포함) + 문진 + 약물코드 힌트 → Gemini 추천 → 실제 처방과 대조.
- **성장 핵심 레지멘 거의 전부 적중**: 성장호르몬(에이큐_G) + GnRH억제(루프린) + 아로마타제억제(아리미덱스) + 비타민D + 수면보조(멜라토닌/5-HTP) 모두 실제 처방과 일치.
- 근거 추론이 임상적으로 타당(E2↑→조기사춘기→억제, IGF-1↑→GH 반응성, VitD 18.2 부족→보충).
- **수면 기록이 없어도** "수면=성장 핵심" 원칙 주입 + 코호트 패턴으로 수면보조 정확히 제안(+ 정직하게 "추가 문진 필요" 표기).
- **ADHD약 제외** 규칙 준수(성장 무관 동반질환).
- **검증된 핵심 교훈**: (1) 약물코드 사전 필수(없으면 GH를 보조제로 오판), (2) 입력 강화(검사+문진+수면)가 적중률 좌우, (3) outcome 라벨 없어도 코호트 패턴+추론으로 의사급 근접, 단 "효과 검증"엔 outcome 분석 필요.
- **데이터 정비 필요**: `prescriptions`에 비약물 항목 혼입(`get_photo`,`get_Review`,`일반 소견서`, 검사코드 등) — 정제 대상.

## 9. 미해결 / 의존성
- **Gemini API 키 만료** — 임베딩 생성·답변 둘 다 막힘. 어떤 단계든 실행 전 키 복구 필요.
- **규제 검토** — 원장 처방 추천이 의료기기(SaMD) 해당 여부 사전 확인 권장. 제시형 프레이밍으로 리스크 완화.
- **데이터 희소성** — 244명. 흔한 프로필 OK, 희귀 조합은 신뢰도 낮음 → 확신도 노출로 대응.
- **PubMed 수집** — E-utilities rate limit(무키 ~3req/s), 초록 XML 파싱, 인종 자동분류 정확도.
- **인구 자동 태깅 한계** — 초록에 population 정보 부족 → Unknown 다수, 핵심만 수동 보정.

## 10. 한 줄 요약
**[국제 논문(초록·인종태그) + 비식별 임상 인사이트(집계통계 + 합성케이스)] → 공유 RAG 브레인 → 입 3개(마케팅/챗봇/원장), 공통 가드레일(의료광고법·인구·개별진단금지·인용/신뢰도·k익명성). 파인튜닝 아님 = RAG 참조. 진료할수록 똑똑해짐.**
